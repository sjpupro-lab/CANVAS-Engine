# CANVAS — 투자자용 개요

> **한 줄 요약**
> CANVAS는 공간 패턴 기반 AI 엔진입니다. 언어를 벡터가 아닌 영상처럼 다뤄, **무한 메모리 · 증분 학습 · 스마트폰에 들어가는 크기**를 동시에 제공합니다.

![CANVAS](main_hero.png)

---

## 1. 문제

오늘날의 프런티어 AI는 단 하나의 구조적 가정 위에 세워져 있습니다: **언어를 고정 크기의 가중치 행렬로 압축한다.**

이 가정은 세 가지 비용을 누적시킵니다.

| 비용 | 결과 |
|---|---|
| **고정 파라미터** | 지식 확장 = 재학습. 모든 업데이트가 풀코스트 이벤트. |
| **제한된 컨텍스트** | 윈도우가 곧 메모리. 그 바깥은 잊힙니다. |
| **불투명한 가중치** | 무엇을 기억하는지 감사·수정·설명할 수 없습니다. |

업계는 이 한계를 우회하기 위해 수천억 달러를 쓰고 있습니다 — 더 큰 모델, 더 긴 컨텍스트, RAG 스택, 벡터 DB. 하지만 이 중 어떤 것도 **기반 구조(substrate)** 자체를 바꾸지는 못합니다.

**병목은 기반 구조 그 자체입니다.**

---

## 2. 인사이트

언어는 공간적 구조를 갖습니다. 비슷한 의미는 가까이 모이고, 다른 의미는 멀어집니다. 이 구조는 보통 벡터 공간으로 투영된 뒤 가중치 안에 눌려 들어갑니다.

**CANVAS는 그 구조를 눈에 보이는 상태로 유지합니다.**

텍스트는 2D 격자 위에 밝기 패턴으로 렌더링됩니다. 언어의 단위는 프레임이 되고, 프레임은 영상처럼 쌓입니다. 메모리는 행렬을 재학습하는 것이 아니라, **프레임을 덧붙여** 자라납니다.

엔진은 패턴 자체를 직접 비교합니다. 임베딩 없이, 어텐션 없이, 재학습 없이.

---

## 3. CANVAS가 여는 것

| | |
|---|---|
| **무한 파라미터** | 새 입력 = 새 프레임. 상한 없이 누적. |
| **무한 컨텍스트** | RAM이 아닌 디스크 용량만이 한계. |
| **해석 가능한 메모리** | 메모리는 시각적 패턴. 엔진이 무엇을 기억하는지 눈으로 확인. |
| **증분 학습** | 새 지식 = 프레임 1장. 재학습도, 파인튜닝도 없음. |
| **엣지 네이티브** | 스마트폰, 웨어러블, 임베디드에서 동작. |

이것은 더 나은 LLM이 아닙니다. **다른 기반**입니다 — LLM을 대체하는 것이 아니라 **보완**합니다.

---

## 4. 기존 LLM과의 비교

```
                    기존 LLM                    CANVAS
  ──────────────────────────────────────────────────────────────
  표현 방식       토큰 → 벡터 → 행렬           격자 위의 패턴
  파라미터        고정 크기 행렬                무한 프레임 스택
  컨텍스트        제한된 윈도우                 디스크 기반 (사실상 무제한)
  해석 가능성     불투명한 가중치               시각적 패턴
  학습 방식       전체 재학습 / 파인튜닝         증분: 프레임 1장
  설치 규모       데이터 센터 수준              임베디드 친화적
```

CANVAS는 경쟁재가 아닌 **상보재**입니다. LLM에게 **없는 메모리 레이어**입니다.

---

## 5. 시장

CANVAS는 세 개의 크고 빠르게 성장하는 시장의 교집합에 있습니다.

- **AI 메모리 / 검색 인프라** — 벡터 DB, RAG 플랫폼, 에이전트 메모리
- **온디바이스 AI** — 엣지 추론, 모바일 AI, 임베디드 어시스턴트
- **해석 가능 / 감사 가능 AI** — 규제 산업, 거버넌스, 안전

각 시장은 이미 수십조 원 규모이고, AI가 챗봇을 넘어 **지속하는 에이전트 · 디바이스 · 규제 워크플로**로 확장되면서 계속 팽창 중입니다.

**웨지(초기 진입점):** 생성보다 패턴의 지속성이 중요한 **검색·메모리·장기 컨텍스트** 워크로드. 에이전트 메모리, 개인 AI, 온디바이스 어시스턴트, 도메인 지식 저장소.

---

## 6. 활용 분야

- **에이전트용 장기 메모리** — 잊지 않는, 벡터 DB가 필요 없는 드롭인 메모리
- **온디바이스 AI** — 클라우드 재학습 없는 증분 업데이트
- **도메인 지식 저장소** — 법률·의료·엔터프라이즈 코퍼스를 한 프레임씩 갱신
- **해석 가능 AI 연구** — 실제로 들여다볼 수 있는 메모리 기반
- **개인 AI** — 사용자와 함께 자라나고, 사용자 디바이스에 남는 메모리

---

## 7. 왜 지금인가

세 가지 흐름이 맞물리고 있습니다.

1. **컨텍스트 윈도우는 물리적·경제적 벽에 부딪혔습니다.** 길이에 선형인 어텐션은 "영원히 기억" 문제를 풀 수 없습니다.
2. **온디바이스 AI가 전략적 우선순위로 올라왔습니다** — OS·통신사·하드웨어 OEM 모두 가벼운 메모리 프리미티브를 필요로 합니다.
3. **해석 가능성·감사 가능성은 연구 과제에서 규제 요구사항으로 이동 중입니다.** 시각적 메모리는 **구조적 답**입니다, 땜질이 아니라.

CANVAS는 처음부터 이 세 가지를 전제로 설계되었습니다.

---

## 8. 모트(Moat)

- **아키텍처 독창성** — 공간 패턴 기반 기반은 트랜스포머·SSM의 변종이 아닌 **전혀 다른 접근**입니다.
- **핵심 IP 비공개 보유** — 공개 저장소는 개요와 데모, **코어 엔진은 별도 비공개 저장소**에서 관리.
- **누적되는 데이터 우위** — 메모리가 증분식이기 때문에 초기 배포가 쌓일수록 검색 품질이 향상되는 패턴 라이브러리가 만들어집니다.
- **풋프린트 우위** — 엣지 동작은 최적화 목표가 아니라 **아키텍처적 속성**입니다. 트랜스포머에 역으로 이식하기가 매우 어렵습니다.

---

## 9. 현황

- **코어 엔진:** 동작하는 구현, 표준 코퍼스에서 벤치마크 완료
- **검증 완료:** 검색, 재현, 증분 메모리 태스크
- **내부 활용:** 장기 컨텍스트 메모리 및 패턴 매칭 워크로드

> 코어 엔진은 **비공개 저장소**에서 관리됩니다. 본 공개 저장소는 개요·컨셉·데모를 담고 있습니다.

## 10. 로드맵

**근거리**
- 파트너 통합용 하드닝된 레퍼런스 구현
- 주요 LLM 스택용 레퍼런스 에이전트 및 메모리 어댑터
- 엣지/모바일 SDK (iOS, Android, 임베디드 리눅스)

**중거리**
- 프로덕션 에이전트 배포를 위한 매니지드 메모리 서비스
- 도메인별 메모리 팩 (법률, 의료, 엔터프라이즈 지식)
- 개발자 플랫폼 — 툴링, 옵저버빌리티, 패턴 에디터

**장거리**
- 온디바이스·에이전트 AI의 **표준 메모리 기반**으로서의 CANVAS
- 메모리 모듈 · 패턴 라이브러리 · 서드파티 통합 생태계
